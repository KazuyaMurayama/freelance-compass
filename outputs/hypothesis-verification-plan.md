# 仮説検証計画 v2 — Claude Codeによる事前検証

**バージョン**: v2.0
**生成日**: 2026-03-16
**策定方法**: 3体の専門AIエージェント議論（検証方法論者 / 連続ソロ起業家 / 認知バイアス監査官）
**目的**: 人間の作業（営業・商談・構築）に入る前に、Claude Codeで検証可能な仮説を先に検証し、事業計画の精度と勝率を上げる
**対象文書**: recommendation.md v29.0 / weekly-execution-plan.md v29.0 / ai-utilization-plan-v7.md

## 0. v1→v2 変更サマリー

### v1で発見された7つの構造的欠陥

3体のエージェント（検証方法論者 / 連続ソロ起業家 / 認知バイアス監査官）の議論により、以下の欠陥を特定し修正した。

| # | v1の問題 | v2の改善 | 提案者 |
|---|---------|---------|--------|
| 1 | ペルソナシミュレーションの確証バイアスリスク（Claude Codeが計画を書き、同じClaude Codeがペルソナを演じる → 「出題者が採点者を兼ねる」構造） | **ブラインド条件の導入**: ペルソナにサービス内容を事前に知らせない段階的開示プロトコル + ペルソナシミュレーションの信頼度に重み付け（デスクリサーチの半分） | 認知バイアス監査官 |
| 2 | 「検証した＝確定した」の錯覚 | **ベイズ更新フレームワーク**: 各仮説に事前確率を設定し、検証ごとに事後確率を更新。手法別の更新幅を制限 | 検証方法論者 |
| 3 | Kill Criteria（事業撤退条件）の不在 | **撤退判断マトリクス**: どの仮説がどう崩れたら事業自体をやめるかを事前定義 | 認知バイアス監査官 |
| 4 | 抽象概念（「AIコンサル」）への反応テストは信頼性が低い | **プロトタイプ検証の追加（新手法F）**: A1診断ツールのサンプル出力+LPモックアップの実物を作り、具体物への反応を取得 | 連続ソロ起業家 |
| 5 | ラウンド間の連携が弱い（「R1がR2に影響する」と書いているが具体的な分岐条件がない） | **条件分岐ツリーの明示**: R1の結果パターン別にR2以降の計画を変更する分岐図を追加 | 検証方法論者 |
| 6 | Go/見直し基準の根拠が曖昧（「直接競合3社以下→Go」は甘すぎる） | **定量的閾値の厳格化**: 各基準に根拠を付記。競合判定は「差別化不能な直接競合」に限定 | 認知バイアス監査官 |
| 7 | 全てが机上検証（「工数3hで作れるか議論する」より「作って計測する」方が速い） | **実測検証の追加（新手法G）**: A1診断ツールを実際に構築し、工数実測値を取得する | 連続ソロ起業家 |

### v2の3つの設計原則

1. **三角測量（Triangulation）**: 1つの仮説を最低2つの異なる手法で検証し、結果が一致した場合のみ「検証済み」とする
2. **ブラインド＋段階的開示**: ペルソナシミュレーションでは、計画の内容を知らない状態から始め、段階的に情報を開示する
3. **定量的確信度管理**: 全仮説にベイズ事前確率を設定し、各検証で事後確率を更新。「何がどれだけ分かったか」を可視化する

## 2. 検証手法の分類（v2: 7手法）

### v1継承手法（A-E）

| 手法 | 名称 | 信頼度の重み | 説明 |
|------|------|-----------|------|
| A | デスクリサーチ | **1.0×** | Web検索・公開データで事実を確認。最もコストが低い |
| B | ペルソナシミュレーション | **0.5×**（v2で引下げ） | AIペルソナによる反応テスト。確証バイアスリスクあり |
| C | 構造分析 | **0.8×** | 数値整合性・工数見積もりの論理検証 |
| D | 敵対的検証 | **1.0×** | プレモーテム・レッドチーム。バイアス排除効果が高い |
| E | 類似事例分析 | **0.9×** | 実在する事例からのベンチマーク |

### v2新規手法（F-G）

| 手法 | 名称 | 信頼度の重み | 説明 |
|------|------|-----------|------|
| **F** | **プロトタイプ検証** | **0.7×** | Claude Codeで実物（診断レポート・LP・提案書）を作成し、具体物への反応を取得 |
| **G** | **実測検証** | **1.2×**（最高信頼度） | Claude Codeで実際に構築し、工数・品質を計測。「作って計る」方式 |

> **信頼度の重みの使い方**: ベイズ更新時に、手法の重みを更新幅に乗算する。例: ペルソナシミュレーション（B: 0.5×）で「Go」結果が出ても、確率の更新幅はデスクリサーチ（A: 1.0×）の半分。

## 3. ベイズ更新フレームワーク（v2新規）

### 3-1. 更新ルール

```
事後確率 = 事前確率 + (基本更新幅 × 手法の信頼度重み × 結果の方向)

基本更新幅:
  - 強い肯定的証拠: +15%
  - 弱い肯定的証拠: +8%
  - 中立/判断不能:   +0%
  - 弱い否定的証拠:  -8%
  - 強い否定的証拠:  -15%

例: H1（事前55%）をペルソナシミュレーション（B: 0.5×）で検証。
    5名中4名が「検討する」と回答（強い肯定）。
    → 55% + (15% × 0.5) = 55% + 7.5% = 62.5%
    （ペルソナの結果だけでは62.5%止まり。確定とは言えない）

例: H1（事前55%）をデスクリサーチ（A: 1.0×）で検証。
    類似サービスが¥10万/月で複数社成立していると確認（強い肯定）。
    → 55% + (15% × 1.0) = 70%
    （デスクリサーチの方が大きく動く）
```

### 3-2. 三角測量のルール

| 検証手法の組合せ | 結果 | 判定 |
|---------------|------|------|
| 2手法以上で肯定 + 否定なし | 事後確率70%以上 | **検証済み（Go）** |
| 1手法で肯定、1手法で中立 | 事後確率60-70% | **追加検証が必要** |
| 1手法以上で否定 | 事後確率50%未満 | **見直し必要** |
| 2手法以上で否定 | 事後確率40%未満 | **Kill Criteria確認** |

### 3-3. 確率ダッシュボード（検証後に更新）

```

                                          40%    50%    60%    70%
H1  WTP          [========|====          ] 55%
H2  最大ジョブ    [========|=            ] 50%
H3  紹介チェーン  [======|               ] 40%    ← CC検証困難
H4  安心PK       [========|======       ] 60%
H5  プラグイン   [=======|=             ] 45%
H6  経営塾       [=====|                ] 35%    ← 最も不確実
H7  品質維持     [=======|=             ] 45%
IH3 競合空白     [========|=            ] 50%    ← 致命的仮説
IH9 Shopify P    [======|               ] 40%    ← 要重点検証
IH16 獲得ペース  [=====|                ] 35%    ← 致命的仮説

```

## 4. Kill Criteria — 事業撤退条件（v2新規）

**原則**: 検証結果によって「事業計画の修正」ではなく「事業自体の中止/根本的ピボット」を判断する基準を事前に定義する。

### 4-1. 即時撤退トリガー

| # | 条件 | 発生する検証 | 撤退理由 |
|---|------|-----------|---------|
| K1 | IH3（競合空白）が事後確率20%未満 | R1-1 | ¥5-15万帯に強力な直接競合が5社以上存在し、差別化が構造的に困難 |
| K2 | H1（WTP）+ IH16（獲得ペース）が両方とも事後確率25%未満 | R1+R2 | 顧客が払わない＋獲得もできない → ビジネスモデル自体が成立しない |
| K3 | R5プレモーテムで「対策不能な構造的リスク」が3件以上発見 | R5-1 | 計画では解決できない根本問題がある |

### 4-2. 根本的ピボットトリガー

| # | 条件 | ピボット先 |
|---|------|----------|
| P1 | IH3が事後40%未満（競合がいるが、差別化は可能） | 価格帯を¥3-8万に下げて「マイクロコンサル」に転換 |
| P2 | H2が事後30%未満（商品PG高速化が最大ジョブではない） | サービス設計を根本変更（実際の最大ジョブに合わせる） |
| P3 | IH9+IH10が両方事後30%未満（チャネルが機能しない） | 営業チャネルを完全再設計（Cold outreach中心に転換） |
| P4 | IH13+IH19が両方事後30%未満（工数が計画の2倍以上） | 提供サービスの範囲を大幅縮小 |

## 5. 検証の実施計画（v2: 6ラウンド構成）

### ラウンド間の条件分岐ツリー

```
R1: デスクリサーチ
├── IH3(競合空白)が事後40%以上 → R2へ進む
├── IH3が事後20-40% → R2を「差別化テスト」に変更（ペルソナに競合比較させる）
└── IH3が事後20%未満 → ★K1撤退判断★

R2: ペルソナシミュレーション（ブラインド方式）
├── H1+H2が両方事後55%以上 → R3へ進む
├── H1またはH2が事後40-55% → R2.5（プロトタイプ検証）を挟む
└── H1+H2が両方事後30%未満 → ★K2撤退判断★

R2.5: プロトタイプ検証（★v2新規★ — R2の結果次第で実施）
├── 具体物への反応が肯定的 → R3へ（H1/H2を上方修正）
└── 具体物への反応も否定的 → ★P2ピボット判断★

R3: 構造分析 + 実測検証
├── 工数見積もり乖離20%以内 → R4へ
└── 工数見積もり乖離50%以上 → ★P4ピボット判断★

R4: 類似事例調査
├── 成功事例あり → R5へ（パターンを反映）
└── 類似モデルが全て失敗 → R5の強度を上げる（徹底的敵対検証）

R5: 敵対的検証
└── 結果を全ラウンドの知見と統合 → 最終判定

R6: 統合判定（★v2新規★）
└── 全仮説の事後確率ダッシュボードに基づき、3ファイルの更新方針を決定
```

### ★v2改善: ブラインド＋段階的開示プロトコル


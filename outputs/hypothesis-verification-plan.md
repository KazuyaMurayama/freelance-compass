# 仮説検証計画 v3 — Claude Codeによる事前検証

**バージョン**: v3.0
**生成日**: 2026-03-17
**策定方法**: v2を基に、新3体のAIエージェント議論（競争戦略アナリスト / リーンスタートアップ実践者 / システム思考家）で改善
**目的**: 人間の作業（営業・商談・構築）に入る前に、Claude Codeで検証可能な仮説を先に検証し、事業計画の精度と勝率を上げる

## 0. v2→v3 変更サマリー

### v2で発見された8つの改善点

新3体のエージェント（競争戦略アナリスト / リーンスタートアップ実践者 / システム思考家）の議論により特定。

| # | v2の問題 | v3の改善 | 提案者 |
|---|---------|---------|--------|

| 2 | 27仮説を独立並列で扱い、仮説間の因果・循環依存を無視 | **4つの戦略クラスターに集約**。仮説間の因果関係を考慮した一括評価 | システム思考 |
| 3 | ベイズ更新が過剰設計（事前確率自体が恣意的。小数点の精度計算は入力の不確実性に見合わない） | **クラスター単位のGreen/Yellow/Red評価にシンプル化**。数値は補助情報に降格 | リーンSU |

### v3の3つの設計原則

1. **クラスター思考**: 27の仮説を4つの戦略クラスターに集約し、個別仮説ではなく「事業の成立条件」単位で判断する
2. **プロトタイプ・ファースト**: まず作る。作ったものを全ての検証で活用する。机上の議論より実物
3. **3日で完了・リアルへ接続**: 事前検証に3日以上かけない。3日で分からないことはリアル検証に回す。事前検証の出口は必ず「最初のリアルアクション」に接続する

## 1. 仮説の戦略クラスター化（★v3新規）

### v2からの最大の構造変更

v2は27仮説を独立並列で管理し、個別にベイズ更新していた。しかし仮説間には因果関係があり、個別検証では全体像を見失う。v3では**4つの戦略クラスター**に集約し、「事業が成立する4つの条件」として一括評価する。

### 1-1. 4つの戦略クラスター

```
┌─────────────────────────────────────────────────┐
│                事業成立の4条件                      │
│                                                   │
│  C1: 市場収益性        C2: チャネル実現性          │
│  「払う人が十分いるか」  「顧客にリーチできるか」    │
│       ↕ 相互依存 ↕                                │
│  C4: 提供価値          C3: 実行可能性              │
│  「価値を感じ続けるか」  「1人で回せるか」          │
└─────────────────────────────────────────────────┘

C1 × C2 = 「売れるか」（Revenue可能性）
C3 × C4 = 「続けられるか」（Sustainability）
4つ全てがGreen/Yellow → Go
1つでもRed → 撤退/ピボット判断
```

### 1-3. IH3（競合空白）の再定義（★v3の核心的変更）

#### v1-v2の問題

v1-v2では「競合がいない＝良い、競合がいる＝悪い」という前提で設計されていた。しかし：

> **競合の存在は市場が存在する証拠である。競合ゼロは「需要がない」可能性が最も高い。**

#### v3の再定義

| 項目 | v2 | **v3** |
|------|-----|--------|
| 仮説の文言 | 「¥5-15万の競合がほぼいない」 | **「¥5-15万の市場で収益を上げられる」** |
| 影響度 | 致命的（競合＝脅威） | **高（競合＝市場の証明 or 脅威の両面）** |
| 検証の問い | 「競合は何社いるか」 | **「市場参加者は収益を上げているか」「差別化余地はあるか」** |

#### 競合の4パターン分類と判断

| パターン | 競合状況 | 収益性 | **判断** |
|---------|---------|--------|---------|
| **A: ブルーオーシャン** | 競合0-1社 | 検証不能 | **要注意**: 市場が存在しない可能性。需要シグナル(R1-5)の結果で判断 |
| **B: 健全な市場** | 競合2-5社 | 少なくとも一部が収益化 | **最も望ましい**: 市場が証明済み。差別化にフォーカス |
| **C: 競争激化** | 競合6-15社 | 収益格差が大きい | **要差別化**: 勝者と敗者がいる。勝つための差別化戦略が必要 |
| **D: レッドオーシャン** | 競合15社以上 + 大手参入 | 価格競争で利益率低下 | **撤退/ピボット**: 1人事業では資本で勝てない |

**v3のKill Criteria**: パターンDかつ差別化戦略を3つ検討しても全て非現実的な場合のみ撤退。パターンA-Cは全てGo（ただしパターン別に戦略を調整）。

## 2. 検証手法の分類（v3: 7手法 — 位置づけ変更）

### 手法一覧

| 手法 | 名称 | **v3の位置づけ** | 説明 |
|------|------|---------------|------|
| A | デスクリサーチ | **事実確認**（高信頼） | Web検索・公開データで事実を確認 |
| B | ペルソナシミュレーション | **★商談準備ツール★**（v3で役割変更） | 仮説の確率更新には使わない。オブジェクション収集・商談練習に特化 |

#### ペルソナシミュレーション（手法B）のリフレーム（★v3最重要変更の1つ）

| 項目 | v2 | **v3** |
|------|-----|--------|
| 目的 | 仮説の確率を更新する | **商談で出るであろう「質問」「不安」「反論」を事前に洗い出す** |
| 信頼度の重み | 0.5×（半分の信頼度でベイズ更新） | **クラスター評価に使わない**。商談準備資料として独立管理 |
| 出力 | 「検討する」と回答したペルソナ数 | **オブジェクション集（想定FAQ）+ 商談ロールプレイ記録** |

## 3. クラスター評価フレームワーク（★v3: ベイズ更新を置換）

### 3-1. 評価方法

v2の27仮説個別ベイズ更新を廃止。代わりに、**4クラスターをGreen/Yellow/Red**の3段階で評価する。

| 評価 | 定義 | アクション |
|------|------|----------|
| **Green** | 複数の検証手法で肯定的証拠。否定的証拠なし | そのまま実行 |
| **Yellow** | 肯定的証拠と否定的証拠が混在。または検証不十分 | 実行するが、リアル検証で重点監視 |
| **Red** | 否定的証拠が優勢。または構造的に成立しない | 撤退/ピボット判断へ |

### 3-2. クラスター別の評価基準

#### C1: 市場収益性

| 評価 | 条件 |
|------|------|
| **Green** | (a)TAMが3,000件以上 AND (b)類似サービスが¥5万以上で成立している事例あり AND (c)需要シグナルあり（ビザスク・ココナラ等で関連相談が存在） |
| **Yellow** | 上記(a)-(c)のうち1つが未確認 or 弱い |
| **Red** | (a)TAM 1,000件未満 OR (b)類似サービスが収益化できていない OR (c)需要痕跡がゼロ |

#### C2: チャネル実現性

| 評価 | 条件 |
|------|------|
| **Green** | 最低2つのチャネルで具体的なリード獲得パスが確認できる |
| **Yellow** | 1つのチャネルのみ確認。または全チャネルが「可能性あり」レベル |
| **Red** | 全チャネルで具体的なリード獲得パスが見つからない |

#### C3: 実行可能性

| 評価 | 条件 |
|------|------|
| **Green** | (a)A1プロトタイプの実測工数が計画の1.5倍以内 AND (b)月次ルーティンの工数積み上げが20h以内 |
| **Yellow** | 工数が計画の1.5-2倍。調整で対応可能 |
| **Red** | 工数が計画の2倍以上。サービス範囲の大幅縮小が必要 |

#### C4: 提供価値

| 評価 | 条件 |
|------|------|
| **Green** | (a)デスクリサーチで「商品PG高速化」が上位課題であると確認 AND (b)プロトタイプ出力の品質が「コンサルとして提案可能」レベル AND (c)類似サービスの解約率が年20%以下 |
| **Yellow** | 上記のうち1つが未確認 or 課題の優先度にブレがある |
| **Red** | ジョブの特定が的外れ AND プロトタイプ品質が不十分 AND 類似サービスの解約率が高い |

### 3-3. 総合判定マトリクス

| C1 | C2 | C3 | C4 | **判定** |
|----|----|----|-----|---------|
| G | G | G | G | **Full Go**: 計画通り実行 |
| G/Y | G/Y | G/Y | G/Y | **Conditional Go**: Yellow箇所を重点監視しつつ実行 |
| R | - | - | - | **撤退判断**: 市場収益性が成立しない |
| - | R | - | - | **チャネル再設計**: 営業戦略を根本変更してリトライ |
| - | - | R | - | **スコープ縮小**: サービス範囲を絞って再設計 |
| - | - | - | R | **サービス再設計**: ジョブの再特定・価値提案の見直し |
| R | R | - | - | **撤退**: 売れない＋届かない → ビジネスモデル不成立 |

## 4. Kill Criteria（v3改訂）

### v2→v3の変更

| 項目 | v2 | **v3** |
|------|-----|--------|
| K1 | IH3（競合空白）が崩壊→撤退 | **廃止**。競合の存在は撤退理由にならない |
| K2 | H1+IH16が両方崩壊→撤退 | **継承（C1 Red判定に統合）** |
| K3 | プレモーテムで対策不能リスク3件以上 | **継承** |
| P1 | IH3が崩壊→マイクロコンサル転換 | **修正**: 競合パターンD（レッドオーシャン）の場合のみピボット検討 |

### v3の撤退条件

| # | 条件 | 判断方法 | 撤退理由 |
|---|------|---------|---------|
| **K1** | **C1（市場収益性）= Red** | TAMが不十分 AND 類似サービスが収益化できていない AND 需要シグナルなし | お金を払う顧客が存在しない |
| **K2** | **C1 = Red AND C2 = Red** | 上記に加え、全チャネルが機能しない | 売れない＋届かない。ビジネスモデル自体が不成立 |
| **K3** | R5プレモーテムで「対策不能な構造的リスク」が3件以上 | 敵対的検証の結果 | 計画では解決できない根本問題 |

### v3のピボット条件

| # | 条件 | ピボット先 |
|---|------|----------|
| **P1** | 競合パターンD（レッドオーシャン）だが市場自体は大きい | 価格帯を下げてマイクロコンサル or ニッチを絞る（例: Shopify×アパレル×月商500-1000万に超特化） |
| **P2** | C4 = Red（H2: 「商品PG高速化」が最大ジョブではない） | 検証で発見された実際の最大ジョブに合わせてサービス再設計 |
| **P3** | C2 = Red（全チャネル不通）だがC1はGreen/Yellow | Cold outreach or コミュニティ営業に完全転換 |
| **P4** | C3 = Red（工数2倍以上） | サービス範囲を大幅縮小（EC限定 or クリニック限定の1業種集中） |

## 6. 検証の限界（v3更新）

| 仮説 | Claude Code検証の限界 | **v3での対処** |
|------|-------------------|-------------|
| H1（WTP）の最終確認 | AIは「お金を払う行動」を再現できない | **リアルアクション・ブリッジ**: ビザスク相談で実際の予算感を確認 |
| H3（紹介チェーン） | 紹介行動は社会関係に依存 | 事前検証対象外。1社目の成功後に観察 |
| H7（品質維持）の実証 | 疲労・ストレスは検証不能 | **Day 0の実測**: 作業のリアリティを体感。パイロット3ヶ月で本格検証 |
| 商談の空気感 | 非言語情報が欠落 | **TC-1のロールプレイ**: 質問と回答を準備。初回Zoom 2-3件で実戦 |
| 競合の実際の強さ | 公開情報では内部事情が分からない | **TA-1の多面調査**: 料金・口コミ・SNSでの存在感から推定。リアル検証では「なぜ前のサービスをやめたか」を商談で確認 |

**v3の基本姿勢**: 事前検証の真の価値は「仮説を確定する」ことではなく、**「最初のリアルアクションで何を確認すべきかを明確にする」**こと。3日の事前検証で全ての仮説に決着をつけようとしない。学びの最大化を目指す。

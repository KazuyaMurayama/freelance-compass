## 0. v6→v7 変更サマリー

v6は売上方程式の「顧客単価」レバーを追加したが、**7つの構造的問題**を抱えていた。

### v6で発見された7つのギャップ

| # | ギャップ | 影響 |
|---|---------|------|
| G1 | 仮説検証(H1-H7)のAI支援がゼロ | v28.0の「仮説未検証の施策は実行しない」原則に違反 |
| G2 | 4層レバレッジの層0/2/3が放置 | 標準M12で層0=¥200k/月の収入源なのにAI活用計画に不在 |
| G3 | Build in Public+99%無料コンテンツのAI自動化がない | v18-19の8種無料コンテンツ戦略が実行不能 |
| G4 | Gate判定の自動支援がない | 5Gate×定量トリガーの判断データが手動集計 |
| G5 | エージェント間データフローが抽象的 | 連携トリガーが未設計で「自動化」が絵に描いた餅 |
| G6 | A10新設でエージェント数が膨張（10体） | 運用複雑化。紹介はA2+A9で実現可能 |
| G7 | 売上シミュレーションがv25.0バイアス排除を反映していない | 楽観的すぎる数値で意思決定を誤るリスク |

### v7の3つの設計原則

1. **Phase連動構築**: エージェントをv28.0のPhase 0→1→2→3に同期して段階構築。「全部事前に作る」をやめる
2. **仮説検証駆動**: H1-H7のGo/Pivot結果を受けてから次の構築を決める。仮説未検証の機能は作らない
3. **10体で役割明確化**: v29.0のA10 ROIストーリーテラーはペルソナ分析に基づく独自の価値（成約率+10-15%、解約防止）を持つため採用。拡張は仮説検証後に段階追加

## 1. 段階構築計画（v7の核心）

### 1-1. 構築の全体像

```

## 3. v7の新機能: BMLサイクル自動化（A7標準機能）

### 3-1. 毎週金曜KPIダッシュボード自動更新

v28.0セクション5C-4で定義されたBuild-Measure-Learn週次サイクル（0.15h=約9分）をA7が自動支援する。

```
A7が毎週金曜に自動実行:

Measure（自動集計）:
  ├── A2からKPIデータ取得（リード数・成約率・NPS・ジョブ完遂率）
  ├── A4から請求データ取得（月次売上・入金状況・解約）
  ├── A9からチャネルデータ取得（Shopify Partner・ビザスク・note PV）
  ├── state/session.jsonからH1-H7の検証進捗を読み取り
  └── 稼働時間の実績集計（月20h制約チェック）

Learn（自動サマリー生成）:
  ├── 前週比の変動をハイライト（例: 「NPS 52→48に低下。要確認」）
  ├── H1-H7の検証進捗を一覧化（未着手/検証中/Go/Pivot）
  ├── Gate判定に必要なデータの充足状況（例: 「Gate 1まであと: NPS 50以上→OK」）

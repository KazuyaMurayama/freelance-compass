# Phase 1: EC事業者のリアルな不満・課題収集

## 調査概要
- 調査日: 2026-03-18
- 調査ソース:
  - Web検索（Google）による業界記事・ブログ記事
  - note.com のEC運営体験記・失敗談
  - Shopifyコミュニティ（日本語フォーラム）の質問傾向
  - ネオマーケティング「EC運用の悩みに関する調査」（2021年、n=743）
  - コレカラ「EC事業者200名アンケート」（2021年、n=200）
  - エートゥジェイ「EC業務の課題と施策に関する調査」（2022年、n=400）
  - HIT-MALL「メーカーEC担当者向けアンケート」（2022年）
  - 各種EC支援企業のブログ・コラム記事（サイバーレコード、ネクストエンジン、スクロール360、BiNDec等）
  - EC運営者の体験談記事（sukeneko.com、UREBAラボ、Yahoo!知恵袋等）

## 1. 時間がかかっている業務（ランキング形式）

### 第1位: 商品登録・ささげ業務（撮影・採寸・原稿）
- **推定所要時間**: 1商品あたり30分〜2時間（撮影・画像加工・説明文作成・スペック入力の合計）。月100SKU追加の場合、月50〜200時間
- **実際の声**:
  - 「商品ページ作成が追いつかない」（ネクストエンジン記事タイトルそのもの）
  - 「取扱商品数が多いと、商品登録の作業量は膨大になる。登録するデータを取り違えないように確認したり、画像を加工したりと、緻密な作業を長時間続ける集中力も求められる」（HIT-MALL）
  - 「商品登録に時間が取られてしまい他の業務にリソースを割けないことが原因で、スタッフの疲弊や最悪の場合は退職につながる」（スクロール360）
  - 複数モール出店時は「モールごとに同じ作業を繰り返す必要がある」（ASPIC）
- **情報ソース**: [ネクストエンジン](https://next-engine.net/ec-blog/ec-business-ai-support/)、[HIT-MALL](https://www.hit-mall.jp/blog/management/column-008.html)、[スクロール360](https://www.scroll360.jp/note/20211102-4933/)

### 第2位: SNS運用・コンテンツ制作
- **推定所要時間**: 1日1〜3時間（投稿作成・撮影・コメント対応・DM管理）。月30〜90時間
- **実際の声**:
  - 「SNS運用は手間がかかるし、全然話題にもならないし...と半ば諦めの気持ちで運用がおざなりになる」（EC運営者の声、ECサイト向け記事より）
  - 「多様化するSNSに都度対応していくことは現実的ではない。限られたリソースの中で、自社の商品や顧客に応じたSNSの取捨選択が求められる」（BiNDec）
  - 「ビジュアル作成から投稿スケジュール管理や担当者管理、コメントやDM管理など煩雑な作業が多い」（Shopify公式ブログ）
  - HIT-MALLの調査では、EC担当者のSNS実施率は100%と定着しているが、その分の業務負荷も常時発生している
- **情報ソース**: [BiNDec](https://bindec.jp/media/555953146194/)、[Shopify日本](https://www.shopify.com/jp/blog/highly-effective-ways-to-promote-a-product)、[HIT-MALL調査](https://www.hit-mall.jp/blog/management/column-060.html)

### 第3位: 集客・広告運用（SEO/リスティング/SNS広告）
- **推定所要時間**: 月20〜60時間（施策立案・クリエイティブ作成・運用・分析レポート）
- **実際の声**:
  - EC事業者200名調査で「最も課題に感じていること」の第1位が「集客・プロモーション」（34.0%）（コレカラ調査）
  - EC業務課題調査で「集客」が56.3%で1位（エートゥジェイ調査、n=400）
  - 「Shopifyを利用して自社でECサイトを運営する場合、ECモールに出店するよりも集客力が低くなるため、自社での集客が必要」（SBペイメントサービス）
  - 「新規顧客獲得に注力し続けることは、広告宣伝費を圧迫するので、あまり賢い選択とはいえない」（BiNDec）
- **情報ソース**: [コレカラ調査](https://corekara.co.jp/contents/sales-up/20211207-2/)、[エートゥジェイ調査](https://netshop.impress.co.jp/node/10468)、[SBペイメントサービス](https://www.sbpayment.jp/support/ec/shopify-build/)

### 第4位: 受注処理・出荷・梱包・配送
- **推定所要時間**: 1日2〜4時間（月商500〜1,500万円規模、1日30〜100件程度の受注の場合）
- **実際の声**:
  - 「受注処理の自動化システムを導入したある事業者では、従来1日あたり2名が4時間かけて行っていた作業が30分程度で完了するようになった」（サイバーレコード）
  - 「発送業務の人手が足りないと、発送遅延が発生し顧客満足度にも影響する」（future-shop）
  - 「注文は24時間入り続け、顧客からの問い合わせも休日関係なく発生するため、いつ休めばいいのか分からないという状況に陥りやすい」（sukeneko.com）
- **情報ソース**: [サイバーレコード](https://www.cyber-records.co.jp/ec-laborshortage/)、[future-shop](https://www.future-shop.jp/magazine/ec-talent-shortage)

### 第5位: 顧客対応（問い合わせ・クレーム・レビュー管理）
- **推定所要時間**: 1日1〜2時間
- **実際の声**:
  - 「顔の見えない顧客からの悪質な口コミの拡散、過度なクレームなど、ECストアを運営する事業者特有のトラブルリスクが顕在化」（損保ジャパン/Shopify記事）
  - 「EC事業者が特に不安と感じているのは、代金回収不能リスクと購入者から過度なクレームを受けるリスク」（ECのミカタ）
  - アパレルECでは「返品、サイズや色の交換などがつきもので、顧客とのコミュニケーションや在庫との連携など作業が煩雑」（Shopifyアパレル事例）
- **情報ソース**: [ECのミカタ](https://ecnomikata.com/ecnews/32496/)、[Shopify公式](https://www.shopify.com/jp/blog/partner-app-sompojapan)

### 第6位: 在庫管理・仕入れ管理
- **推定所要時間**: 月10〜30時間
- **実際の声**:
  - 「手動での在庫更新によるタイムラグは、売り越しや販売機会損失の原因となり、顧客からの信用を大きく損なう」（凸版印刷/TOPPAN記事）
  - 「複数のモールで運営している場合は在庫連携が課題」（ネクストエンジン）
  - 「ブランドごとに別々のECサイトを運営していたため、商品情報の更新や売上データ集計に多大な手間がかかっていた」（LAC WATCH Shopify事例）
- **情報ソース**: [TOPPAN](https://solution.toppan.co.jp/bx/contents/ec_site02.html)、[LAC WATCH](https://www.lac.co.jp/lacwatch/service/20251003_004509.html)

### 第7位: データ分析・レポーティング
- **推定所要時間**: 月5〜15時間
- **実際の声**:
  - EC業務課題調査で「顧客分析ができていない」が21.0%（エートゥジェイ調査）
  - CRMツール未導入率が51.2%（同調査）
  - CRM施策実施者のうち47.3%が「不十分」と回答、主因は「人的リソース不足」（ネオマーケティング調査）
- **情報ソース**: [エートゥジェイ調査](https://netshop.impress.co.jp/node/10468)、[ネオマーケティング調査](https://corp.neo-m.jp/report/investigation/itmedia_029_e-commerce-problem)

### 補足: 全体の時間負担感
- ネオマーケティング調査（n=517）によると、**EC運用にかかる時間・工数を「多い」と感じている担当者は48.2%**（約半数）
- 「運用リソース不足」が課題の第2位（14.5%）

## 2. お金を払ってもいいと思える領域

### 実際に課金しているツール・サービス

| カテゴリ | 主なツール/サービス | 月額相場 |
|---------|-------------------|---------|
| カートシステム | Shopify（ベーシック月額約4,000円〜、スタンダード約12,000円）、makeshop、カラーミーショップ | 3,000〜100,000円 |
| 受注・在庫一元管理 | ネクストエンジン、CROSS MALL、助ネコ | 15,000〜30,000円〜 |
| メール配信/CRM/MA | Klaviyo、Omnisend、HubSpot | 5,000〜50,000円 |
| 決済サービス | Shopify Payments、SBペイメントサービス | 売上の3〜5% |
| 広告運用 | Google Ads、Meta広告（自社運用 or 代理店） | 広告費の20%が代理店手数料の相場 |
| 物流代行 | スクロール360、オープンロジ等 | 1件300〜2,000円+月額基本料 |
| 商品登録代行 | 各種BPO企業 | 1商品あたり100〜1,500円 |

### 外注している業務と相場

| 外注業務 | 費用相場 |
|---------|---------|
| EC運営代行（全体） | 月額30万円以上 |
| EC運営代行（一部業務） | 月額5〜10万円 |
| ECコンサルティング | 月額5〜50万円 |
| サイト構築・リニューアル | 50〜300万円（一括） |
| 商品撮影（プロ） | 1カット500〜5,000円 |
| SNS運用代行 | 月額10〜30万円 |
| 広告運用代行 | 広告費の20%+月額固定費 |

- ネオマーケティング調査によると、外部パートナー活用率は**約6割**
- 外部委託先トップは「サイト構築」（6割以上）、次いで「Webマーケティング」「市場調査」（各3割以上）
- 外注のネックは「コストの高さ」が最多（コレカラ調査）

### 「これがあれば払う」という潜在需要

1. **商品登録の完全自動化**: 撮影からテキスト生成・登録までワンストップで完結するサービス。現状のShopify Magic（AI商品説明文生成）は「確認・微調整が必要」で完全自動化には至っていない
2. **SNS投稿の自動生成・スケジュール管理**: 商品情報から自動でSNS投稿を生成し、最適タイミングで配信
3. **低コストのEC運営コンサル**: 月額5万円以下で実務的なアドバイスが得られるサービス。現状は「月額5万円〜」が最低ラインだが、スモールビジネスには高い
4. **複数モールの在庫・受注一元管理**: 既にネクストエンジン等が存在するが、月額3万円〜と中小規模には負担
5. **顧客分析・CRM自動化**: CRMツール未導入率51.2%（エートゥジェイ調査）であり、「使いたいが導入・運用が難しい」層が大きい
6. **リピーター獲得の仕組み化**: EC業務課題で「リピート購入」が40.0%で2位（エートゥジェイ調査）

## 3. 失敗・トラウマ事例

### コンサル・ツール導入の失敗パターン

#### パターン1: プラットフォーム選択の誤り
- 「Amazonコンサルを依頼したが成果が出ず、実は楽天向けの需要が大きかった」（goat-ec.com失敗事例集）
- コンサル側の得意分野が優先され、事業者の商材に合わないプラットフォームを推奨される
- **教訓**: コンサルの得意領域と自社商材の適性を事前に確認すべき

#### パターン2: 過大なシミュレーションによる期待値ズレ
- 「3か月で売上2倍」等の根拠不十分な数字を提示され、実績が**シミュレーションの20%程度**に終わるケースも（goat-ec.com）
- 成果が出ないと「広告予算や制作物に問題がある」と追加プランを提示される
- **教訓**: 根拠のない数値約束をする業者は危険信号

#### パターン3: 担当者変更による品質低下
- 華やかな経歴の営業担当が契約後、経験の浅いスタッフに引き継がれる（goat-ec.com）
- 途中で複数回担当者が変わり、引き継ぎが不十分になる
- **教訓**: 実務担当者が誰かを契約前に確認

#### パターン4: 成果報酬型の罠
- 売上連動型報酬で「過度な値下げやクーポン施策により利益率が減少」（goat-ec.com）
- 売上は増加しコンサル報酬も増加するが、**1年程度で店舗が退店に追い込まれる**ケースも
- **教訓**: 売上だけでなく利益率を含めた契約設計が必要

#### パターン5: マッチングサービスの中抜き問題
- 事業者が月額30万円支払うも、実際のEC人材には**15万円しか支払われていない**（goat-ec.com）
- マッチング業者がマージンを半分抜くケースも存在

#### パターン6: 広告費の自転車操業化
- PMF（プロダクト・マーケット・フィット）が不足した状態での広告投下により「自転車操業化」（goat-ec.com）
- 広告費20%手数料で「成果が出ず、コストばかり増える」事態

### 信頼獲得のハードル

EC事業者がコンサルやツール導入に慎重になる理由:
1. **過去の失敗体験**: 上記パターンの被害経験者が口コミで警戒を広める
2. **成果の可視化が困難**: ECの売上要因は複合的であり、特定施策の効果を切り分けにくい
3. **EC業界は2〜3年で成果**: 基本的にEC事業として成り立つまで2〜3年はかかるとされ（ソフトコム）、短期成果を約束する業者への不信感
4. **情報の非対称性**: EC事業者側にリテラシーが不足していると、提案内容の妥当性を判断できない
- **情報ソース**: [goat-ec.com](https://goat-ec.com/ec-consulting-fault-cases/)、[ソフトコム](https://www.soft-com.co.jp/column/814/)、[丹誠リーガルオフィス](https://tansei-legaloffice.jp/tansei-blog/consumer-damage-fraud/fukugyousagi/2049/)

## 4. 不安・将来懸念

### EC事業者が感じている構造的な不安

#### 1. プラットフォーム依存リスク
- モール出店者は「手数料負担や価格競争で利益率確保が課題」（eコマースコンバージョンラボ）
- AmazonとShopifyのエコシステム競争の行方次第で、出店事業者が「コンテンツフォーマットのオプション、顧客データ、ストアポリシーのコントロールを維持できなくなる可能性」（ネットショップ担当者フォーラム）
- Shopifyの料金改定・仕様変更に振り回されるリスク

#### 2. 競合増加と市場の飽和
- EC市場は成長しているが、それだけに事業者間の競争も激化（SBペイメントサービス）
- コロナ禍以降に多くの企業が新規参入し、「限られた人材の奪い合い」も発生（future-shop）
- 「作ったけれど売れない」「システムが使いにくく運用が回らない」が最も多い失敗（発注ラウンジ）

#### 3. 広告費高騰と集客効率の低下
- 新規顧客獲得コスト（CPA）の上昇傾向
- Cookieベースのマーケティングの終焉により、パーソナライズされた広告配信が困難に（Shopify公式）
- 企業アカウントのSNS投稿が「生活者のタイムライン上に表示されづらくなっている」（ガイアックス2024年レポート）

#### 4. 人材確保の困難
- EC人材の需給不均衡が深刻化（future-shop、TOKYO GATE）
- 「1人当たりの担当業務範囲が広くなり、業務量の多さや求められるスキルの幅広さが業務負荷を高め、結果として離職率を高める要因」（future-shop）
- 「朝9時出勤の深夜3時上がり」「鬱のような症状が現れて転職を決意」等のブラック化事例（体験談記事）

#### 5. セキュリティ・不正利用リスク
- クレジットカードの不正利用、チャージバック被害（損保ジャパン/Shopify）
- なりすまし偽サイト・偽ショッピングサイトの増加（ネットショップ担当者フォーラム）
- Shopifyの急速な普及がサイバー攻撃のターゲットにもなりやすい状況を生んでいる（セキュリティ専門記事）

#### 6. 売上の季節変動・不安定さ
- 繁忙期と閑散期の差が激しく、人員配置・在庫計画が困難
- 「リスクも大きくなり、特に広告の管理、在庫の管理、運用（人）の管理をしっかり押さえておかないと、市場の変化によっては一気に事業が傾く可能性」（ECMJ）

- **情報ソース**: [ネットショップ担当者フォーラム](https://netshop.impress.co.jp/node/8645)、[future-shop](https://www.future-shop.jp/magazine/ec-talent-shortage)、[ECMJ](https://www.ecmj.co.jp/)、[損保ジャパン/Shopify](https://www.shopify.com/jp/blog/partner-app-sompojapan)

## 5. 発見・インサイト

### 予想外だった発見

1. **EC運営者の約半数が「時間・工数が多すぎる」と感じている**（ネオマーケティング調査で48.2%）。これは「なんとなく大変」ではなく、定量的に確認された事実である

2. **CRMツール未導入率が51.2%と過半数**（エートゥジェイ調査）。リピート購入が課題の2位（40.0%）にランクインしているにもかかわらず、対策ツールを導入していない事業者が過半数。「必要性は感じているが導入・運用できない」というギャップが大きい

3. **外部パートナー活用率は約6割だが、最も多い委託先は「サイト構築」**（ネオマーケティング調査）。日常的な運営業務（商品登録、SNS運用、顧客対応等）の外注はまだ一般的でない。ここに潜在市場がある

4. **ECコンサルの失敗事例が非常に具体的かつ深刻**。「シミュレーションの20%程度の成果」「1年で退店」「マージン50%中抜き」など、業界構造的な問題が存在する。この不信感が新規サービス導入の障壁になっている

5. **「出荷・配送」「問い合わせ対応」は課題と感じている人が少ない**（コレカラ調査）。物流は外注ソリューションが成熟しており、痛みが解消されつつある。一方、「集客」「商品登録」「SNS運用」は依然として痛みが大きい

6. **Shopifyの英語情報中心・日本語サポートの弱さ**が中小事業者にとって大きな壁。メールサポートは返答まで5〜10営業日かかることも（Shopifyガイド記事）

7. **AI活用の期待値は高いが実装ギャップが大きい**。Shopify MagicやSidekickは存在するが、「AI生成されたテキストには誤情報が含まれる場合がある」（Shopify公式注意喚起）ため、結局人間の確認作業が必要。「AIで全自動」への期待と現実の乖離が不満の種になりうる

### 課題の深刻度ランキング（支払意欲との相関）

| 順位 | 課題 | 深刻度 | 支払意欲 | 備考 |
|-----|------|-------|---------|------|
| 1 | 集客・新規顧客獲得 | ★★★★★ | ★★★★★ | 最大の課題であり、最も予算を割いている領域。ただし広告費の費用対効果への不満も大きい |
| 2 | 商品登録・ささげ業務 | ★★★★★ | ★★★☆☆ | 深刻だが「お金を払って解決する」文化がまだ根付いていない。代行相場（1商品100〜1,500円）は認知されつつある |
| 3 | SNS運用・コンテンツ制作 | ★★★★☆ | ★★★☆☆ | 実施率100%だが成果が見えにくく、外注判断が難しい。月額10〜30万円の代行は中小には高い |
| 4 | リピーター獲得・CRM | ★★★★☆ | ★★★★☆ | 課題認識は高い（40.0%が課題と回答）が、ツール未導入率51.2%。導入ハードルを下げれば支払意欲は高い |
| 5 | 在庫管理・受注処理 | ★★★☆☆ | ★★★★☆ | 既存ツール（ネクストエンジン等）が成熟。月商500万円超では投資対効果を感じやすい |
| 6 | データ分析・レポーティング | ★★★☆☆ | ★★☆☆☆ | 必要性は理解されているが、「分析して何をすればいいか分からない」という二重の壁がある |
| 7 | セキュリティ・不正対策 | ★★☆☆☆ | ★★☆☆☆ | 被害に遭うまで対策しない傾向。保険型商品（損保ジャパンアプリ）は新しいアプローチ |

### フリーランスとしての参入機会（総合考察）

月商500〜1,500万円規模のEC事業者が最も痛みを感じているのは「集客」と「商品登録」だが、集客領域は既にレッドオーシャン（広告代理店・ECコンサルが乱立）。一方、以下の領域は**競合が少なく支払意欲もある**ニッチ：

1. **商品登録のAI半自動化支援**: Shopify Magic等の既存AIツールを組み合わせ、「撮影データを渡せば商品ページが出来上がる」仕組みを構築・運用する代行。1商品あたりの単価設定で月10〜20時間の稼働が可能
2. **CRM/リピーター施策の導入・運用代行**: 未導入率51.2%の巨大ギャップ。Klaviyo等のツール設定〜初期シナリオ構築を月額5〜10万円で提供
3. **EC運営の部分最適化コンサル**: フルコンサル（月額30万円〜）ではなく、特定課題（例: 商品ページ改善、メール施策）に絞った月額3〜10万円のスポット支援。業界のコンサル不信を逆手に取り、小さく始めて実績で信頼を積む

# Phase 3: Shopify API技術制約確認＋プロトタイプ構想

> 調査日: 2026-03-18
> 調査対象: Shopify Admin API / Storefront API / 関連エコシステム

## 1. Shopify API概要

### 1.1 Admin API

Shopify Admin APIはマーチャント向けのバックエンド操作APIであり、商品・注文・顧客・在庫・フルフィルメント等のデータに対してフルのCRUD操作が可能。

**重要な移行情報:**
- REST Admin APIは2024年10月1日をもって**レガシーAPI**に指定
- 2025年4月1日以降、新規Public AppはGraphQL Admin APIのみで構築が必須
- 既存アプリのREST利用は当面継続可能だが、完全移行期限が別途設定される予定

**主要リソース:**

| リソース | 取得可能なデータ | 更新可能な範囲 | 必要なスコープ |
|----------|-----------------|---------------|---------------|
| **Products** | title, descriptionHtml, vendor, product_type, tags, variants, images, SEOメタ, status, handle | 全フィールド更新可能。`productUpdate`ミューテーションで一括更新も可 | `read_products`, `write_products` |
| **Orders** | order_number, financial_status, fulfillment_status, line_items, customer, shipping_address, billing_address, total_price, subtotal_price, tax, discount, created_at, refunds | ノート・タグ・配送先の更新、キャンセル、返品処理 | `read_orders`, `write_orders` |
| **Customers** | email, first_name, last_name, orders_count, total_spent, tags, addresses, created_at, accepts_marketing, `lifetimeSpendAndOrderCount`(GraphQL) | 顧客情報の更新、タグ付け、マーケティング同意状態 | `read_customers`, `write_customers` |
| **Inventory** | InventoryItem(SKU, tracked, 関税情報), InventoryLevel(available, on_hand, incoming, committed, reserved等の数量状態) | 在庫の有効化、数量調整、状態間移動（available↔reserved等） | `read_inventory`, `write_inventory` |

**Ordersの重要な制限:**
- デフォルトでは**直近60日分**の注文のみ取得可能
- 全期間のアクセスには`read_all_orders`スコープが必要（Shopifyの審査が必要）

**Inventoryの重要な制限:**
- `committed`状態の在庫はAPI経由で調整不可（注文の作成・フルフィルメントでのみ変動）
- API version 2026-04以降、冪等性キーが必須

### 1.2 Storefront API

Storefront APIは**顧客向け（フロントエンド）体験**を構築するためのGraphQL API。

**できること:**
- 商品カタログの参照（商品、コレクション、バリアント）
- カートの作成・管理
- チェックアウトの開始
- 顧客認証（ログイン・登録）
- コンテンツ（ブログ、ページ）の取得
- ヘッドレスコマース、モバイルアプリ、IoT向けストアフロント構築

**できないこと:**
- 注文管理・フルフィルメント操作
- 在庫数の更新
- 顧客データの管理的操作
- 商品の作成・更新・削除
- 価格ルールやディスカウントの管理
- ストア設定の変更

**Admin APIとの主な違い:**

| 項目 | Admin API | Storefront API |
|------|-----------|----------------|
| アクセス | サーバーサイド専用（秘密鍵） | クライアントサイド安全（公開可） |
| データ操作 | フルCRUD | 主にRead + カート操作 |
| レート制限 | コストベース（後述） | リクエスト数制限なし（バイヤーIP単位） |
| ユースケース | バックオフィス管理 | カスタムストアフロント |
| API形式 | GraphQL（推奨）+ REST（レガシー） | GraphQLのみ |
| ID形式 | `gid://shopify/Product/1` | Base64エンコード |

### 1.3 Rate Limits & 制約

#### GraphQL Admin API（推奨）

| プラン | バケットサイズ | リストアレート |
|--------|-------------|--------------|
| Standard | 2,000ポイント | 50ポイント/秒 |
| Advanced | 4,000ポイント | 100ポイント/秒（推定） |
| Shopify Plus | 20,000ポイント | 500ポイント/秒（推定） |

- **1リクエストあたりの最大コスト**: 1,000ポイント
- **ミューテーション**: 10ポイント/回
- **クエリ（オブジェクト取得）**: 1ポイント/オブジェクト
- **バルクオペレーション**: レート制限の対象外（大量データ処理用）
- **在庫調整帯域**: 最大500アイテム/秒、180万調整/時間

#### REST Admin API（レガシー）

| プラン | バケットサイズ | リークレート |
|--------|-------------|------------|
| Standard | 40リクエスト | 2リクエスト/秒 |
| Shopify Plus | 400リクエスト | 20リクエスト/秒 |

#### Storefront API
- リクエスト数の制限なし（バイヤーIP単位のレート制限）
- トークンレスアクセスの場合、クエリ複雑度上限1,000

#### その他の制限
- 商品バリアントが50,000を超えるストアでは、1日あたり1,000バリアントまでの追加制限（Shopify Plusは対象外）
- APIコール自体は**無料**（全プランで追加料金なし）

### 1.4 認証方式

| 方式 | 特徴 | 2026年以降の状況 |
|------|------|-----------------|
| **Custom App（Admin直接作成）** | シンプルな固定トークン。単一ストア向け | 2026年1月1日以降、Admin経由での新規作成不可。Dev Dashboard経由に移行 |
| **Custom App（Dev Dashboard作成）** | クライアントクレデンシャルグラント方式。トークン有効期限24時間 | 今後の標準。単一ストア向けカスタム連携に最適 |
| **Public App** | OAuth 2.0フロー。複数ストアへの配布可能 | 2025年4月以降、GraphQL必須。App Storeレビューが必要 |

**EC事業者向けサービスとしての推奨:**
- **初期MVP（特定クライアント向け）**: Custom App（Dev Dashboard作成）が最適
- **スケール時（複数クライアント対応）**: Public Appへの移行を検討

## 2. ユースケース別 技術的実現性評価

| ユースケース | 実現性 | 必要なAPI/サービス | 工数目安（Claude Code活用） | 備考 |
|-------------|--------|-------------------|--------------------------|------|
| a) 商品説明文の自動生成/改善 | ★★★★★ | Admin API (Products) + Claude API | 2-3日 | 最もROI高い。即座に価値を示せる |
| b) 売上データ分析ダッシュボード | ★★★★☆ | Admin API (Orders) + GA4 | 5-7日 | Orders APIの60日制限に注意 |
| c) 顧客セグメント分析（RFM/LTV） | ★★★★☆ | Admin API (Customers, Orders) | 3-5日 | `read_all_orders`スコープが必要 |
| d) 在庫最適化アラート | ★★★☆☆ | Admin API (Inventory) + Webhooks | 3-5日 | 在庫回転率算出にはOrders連携必須 |
| e) 広告パフォーマンス分析 | ★★☆☆☆ | Meta CAPI, Google Ads API, Shopify外部 | 7-10日 | Shopify API単体では不十分。外部API連携必須 |
| f) チャットボット/FAQ自動生成 | ★★★☆☆ | Storefront API + Claude API | 5-7日 | Shopify Inboxは外部APIなし。独自構築が必要 |

### a) 商品説明文の自動生成/改善

**技術的実現方法:**

1. **データ取得**: Admin GraphQL APIで`products`クエリにより商品情報（title, descriptionHtml, images, variants, product_type, tags）を取得
2. **AI処理**: Claude APIに商品データを渡し、SEO最適化された説明文を生成
3. **画像認識**: Claude Vision APIで商品画像を解析し、視覚的特徴を説明文に反映
4. **一括更新**: `productUpdate`ミューテーションで`descriptionHtml`を更新

**一括更新の方法:**
- 少量（〜100商品）: 個別の`productUpdate`ミューテーションをループ実行（レート制限内で十分対応可能）
- 大量（100商品以上）: バルクオペレーション（JSONL形式でアップロード → `bulkOperationRunMutation` → Webhook完了通知）

**実装の流れ:**
```
商品一覧取得（GraphQL） → 商品画像DL → Claude Vision解析 → 説明文生成 → descriptionHtml更新
```

**注意点:**
- バルクミューテーションはJSONLファイルの作成・アップロード・完了待ちのパイプラインが必要
- 商品画像はURL経由でClaude Vision APIに直接渡せるため、ダウンロード不要
- 更新前にプレビュー機能を設けてクライアント確認を挟むのが実運用上必須

### b) 売上データ分析ダッシュボード

**Orders APIから取得可能な分析指標:**
- 売上推移（日次/週次/月次）: `created_at`, `total_price`
- 商品別売上: `line_items`の`price`, `quantity`
- 平均注文額（AOV）: `total_price`の平均
- 顧客別売上: `customer`フィールドとの紐付け
- 地域別売上: `shipping_address`の`country`, `province`
- 割引利用率: `discount_codes`, `total_discounts`
- 返品・返金率: `refunds`フィールド
- フルフィルメント状況: `fulfillment_status`

**GA4連携の方法:**
- **Shopify Pixel（Web Pixel API）**: Shopify標準のイベントトラッキング基盤。カスタムPixelとしてGA4/GTMを設定可能
- **Google & YouTube Channel App**: 最も簡易な連携方法。GA4プロパティを接続するだけ
- **カスタムPixel + GTM**: `analytics.subscribe()`でShopifyイベントをGTMデータレイヤーに送信。高度なカスタマイズ可能
- **注意**: カスタムPixelはサンドボックス化されたiFrame内で動作するため、一部の自動検出機能が制限される

**ROAS算出に必要なデータソース:**
- 売上データ: Shopify Orders API
- 広告費データ: Meta Ads API / Google Ads API（Shopify外部）
- ROAS = 売上 ÷ 広告費（クロスプラットフォームの集計が必要）

**Shopify API単体でのROAS算出は不可**。広告プラットフォームのAPIと組み合わせる必要がある。

### c) 顧客セグメント分析（RFM/LTV）

**RFM分析の実現可能性: 高い**

必要なデータはすべてAdmin APIから取得可能:
- **Recency**: 各顧客の最終注文日（`orders`クエリで`sortKey: CREATED_AT`）
- **Frequency**: 顧客の`orders_count`、またはOrders APIで注文回数を集計
- **Monetary**: `total_spent`フィールド、またはGraphQLの`lifetimeSpendAndOrderCount`

**LTV自動算出の実装:**
```
LTV = AOV × 購入頻度 × 顧客寿命
```
- AOV: 顧客の全注文の`total_price`平均
- 購入頻度: 一定期間内の注文回数
- 顧客寿命: 初回注文〜最終注文の期間

**実装上の注意:**
- `read_all_orders`スコープが必須（60日制限を超えるため）。Shopifyによる審査あり
- 初回は全注文データのバルク取得（`bulkOperationRunQuery`）が効率的
- 以降はWebhook（`orders/create`）でリアルタイム更新
- セグメント結果をShopify顧客タグに反映可能（`customerUpdate`ミューテーション）

### d) 在庫最適化アラート

**在庫回転率の自動計算:**
- 在庫回転率 = 売上原価 ÷ 平均在庫
- 必要データ: InventoryLevel（現在庫）+ Orders（販売数量）+ Product（原価情報）
- Shopify APIでは売上原価（COGS）が直接取得しにくい場合がある。`inventoryItem`の`cost`フィールドで仕入原価は取得可能

**発注タイミング予測:**
- 過去の販売速度（日次/週次の販売数量）を計算
- 現在庫 ÷ 平均日次販売数 = 在庫日数
- リードタイム（手動設定）と比較してアラート発火

**Webhook活用:**
- `inventory_levels/update`: 在庫変動時にリアルタイム通知
- `orders/create`: 新規注文時に在庫影響を即時計算

**制約:**
- 在庫の`committed`状態はAPIで操作不可
- リードタイムや発注ロットサイズはShopify APIに存在しないため、別途マスタデータとして管理が必要

### e) 広告パフォーマンス分析

**Shopify APIだけでは実現困難**。以下の外部連携が必須:

| データソース | 取得方法 | 取得できるデータ |
|------------|---------|----------------|
| **Meta Ads** | Meta Marketing API / Conversions API | 広告費、インプレッション、クリック、コンバージョン |
| **Google Ads** | Google Ads API | 広告費、クリック、CPC、コンバージョン |
| **Shopify** | Orders API | 実売上、注文、顧客データ |

**Shopify側の連携機能:**
- **Shopify Audiences**: Shopify Plus限定。顧客行動データからMeta/Google向けの高精度オーディエンスを自動生成
- **Meta Conversions API（CAPI）**: サーバーサイドでコンバージョンデータをMetaに送信。Pixelと併用で精度向上
- **Google Data Manager連携**: サーバー間通信でコンバージョンシグナルを回復（現時点では「Checkout complete」イベントのみ対応）

**実装の現実的なアプローチ:**
- Shopify Orders APIで売上データ取得 + 各広告プラットフォームAPIで広告費取得 → クロス集計
- 初期MVPではCSVインポート（手動）でも実用的

### f) チャットボット/FAQ自動生成

**過去の問い合わせデータ取得:**
- **Shopify Inbox**: 外部向けAPIは提供されていない。データのエクスポート機能も限定的
- **代替手段**: メール問い合わせ（Gmailなど）のAPI連携、またはZendesk/Freshdesk等のヘルプデスクツールからのデータ取得

**FAQ自動生成の実現方法:**
1. 商品データ（Products API）+ ストアポリシー（手動入力 or Shop API）をナレッジベースとして構築
2. Claude APIで質問応答システムを構築
3. Storefront APIまたはShopifyテーマのApp Embed Blockとしてチャットウィジェットを配置

**Shopify App Embed Block:**
- テーマに組み込めるウィジェットとして、チャットボットUIをShopifyストアに追加可能
- Shopify App Extensionsの仕組みを利用

**制約:**
- Shopify Inboxとの直接API連携は不可
- 独自チャットボットを構築する場合、会話ログの保存・管理は自前で実装が必要
- 顧客の注文状況参照にはAdmin API連携（サーバーサイド）が必要

### 3.2 Claude API + Shopify API 連携アーキテクチャ

```
┌─────────────┐     ┌──────────────────┐     ┌───────────────┐
│  クライアント  │────▶│  Next.js API     │────▶│ Shopify Admin │
│  (ブラウザ)   │◀────│  Routes          │◀────│ GraphQL API   │
└─────────────┘     │                  │     └───────────────┘
                    │  ┌────────────┐  │
                    │  │ Claude API │  │     ┌───────────────┐
                    │  │ (Anthropic)│  │────▶│  PostgreSQL   │
                    │  └────────────┘  │◀────│  (Supabase)   │
                    └──────────────────┘     └───────────────┘
                           │
                    ┌──────┴──────┐
                    │  Webhooks   │
                    │ (orders/    │
                    │  inventory) │
                    └─────────────┘
```

**データフロー:**
1. クライアントがNext.js API Routeにリクエスト
2. API RouteがShopify Admin GraphQL APIからデータ取得
3. 取得データをClaude APIに送信（説明文生成、分析等）
4. 結果をDBに保存し、クライアントに返却
5. Webhookで注文・在庫変動をリアルタイム受信し、DB更新

・日々の運営で最も手間がかかっている作業は何ですか？
・「これが自動化できたら嬉しい」と思う業務はありますか？
・外部に相談したいが、誰に聞けばいいかわからない課題はありますか？

営業目的ではなく、純粋にEC事業者の課題を理解したいという動機です。
お気軽にコメントいただけると嬉しいです。
```

**Step 4: 個別コンタクト**
1. コメントをくれた方、または活発に投稿している方のプロフィールを確認
2. コミュニティ内のDM機能（「メッセージを送る」）は限定的なため、プロフィールにSNSリンクがあればそこ経由でコンタクト
3. スレッド内で「もしよろしければ、もう少し詳しくお話を伺えませんか？」と返信し、連絡手段（メール・X等）を提示

### 2a. ビザスク（visasq）

#### サービス概要
ビザスクは、業界経験者（アドバイザー/エキスパート）に1時間単位でスポットコンサルができるマッチングプラットフォーム。登録アドバイザー数は国内最大級の60万人超。

#### 料金体系

| プラン | 1回あたり費用 | 特徴 |
|--------|-------------|------|
| ビザスクlite（セルフマッチング） | 15,000〜30,000円/1時間 | 自分でアドバイザーを検索・指名 |
| ビザスク interview（フルサポート） | 約100,000円/1時間 | 専任スタッフがマッチング |

**今回の用途では「ビザスクlite」を使用。1件あたり15,000〜20,000円が目安。**

#### 登録方法（10分）
1. https://visasq.co.jp にアクセス
2. 「無料で登録する」をクリック
3. メールアドレス、またはFacebook/LinkedInアカウントで登録
4. 氏名・所属・依頼目的を入力
5. 依頼者として登録完了（登録自体は無料）

#### EC事業者にインタビュー依頼する手順

**方法1: 指名相談**
1. ログイン後、「エキスパートを探す」ページへ
2. キーワード: `Shopify EC運営 月商` `ECサイト 運営責任者`
3. 検索結果からプロフィールを確認
4. 適切なエキスパートに「相談する」ボタンからメッセージを送信
5. 謝礼金額・日時・実施方法（Web会議）を相談

**方法2: 公募**
1. 「案件を公募する」から新規案件を作成
2. タイトル例：「Shopify利用EC事業者の方に運営課題についてヒアリング（30分）」
3. 依頼内容に詳細を記載
4. 謝礼金額を設定（15,000円程度）
5. エキスパートからの応募を待つ

#### メリット / デメリット

| メリット | デメリット |
|---------|----------|
| 確実にインタビューが実施できる | 1件15,000〜30,000円とコストが高い |
| 相手の経歴・専門が事前にわかる | 月商規模の指定が難しい場合がある |
| プラットフォームの信頼性が高い | 3〜5名で45,000〜100,000円の予算が必要 |
| 日程調整がスムーズ | 予算「月数千円」には収まらない |

#### 予算が限られる場合の対策
- 謝礼を最低ライン（15,000円）に設定
- 30分のインタビューとして半額交渉を試みる（応じてもらえない場合もある）
- **1〜2名はビザスク、残りは無料チャネルで確保**するハイブリッド戦略がおすすめ

## 1. リサーチから浮かび上がった「本当の課題」の構造

### 課題の3層構造

Phase 1の調査結果を分析すると、EC事業者の課題は以下の3層に整理できる：

```
【表層】 時間が足りない（48.2%が「工数多すぎ」）
    ↓
【中層】 何から手をつければいいか分からない（ツール乱立、情報過多）
    ↓
【深層】 「やったことが正しかったか」を検証できない（効果測定の欠如）
```

**重要な発見**: EC事業者は「時間がない」と言うが、本質的な痛みは**「限られた時間で何をすべきか分からない」**こと。商品登録に月50〜200時間かけている事業者がいる一方で、その商品ページの改善が売上にどう影響したかを測定できていない。

### 支払意欲の構造

| 課題領域 | 深刻度 | 支払意欲 | ギャップ | 参入機会 |
|---------|--------|---------|---------|---------|
| 集客・広告 | ★★★★★ | ★★★★★ | 小（既にレッドオーシャン） | **低** |
| 商品登録・ささげ | ★★★★★ | ★★★☆☆ | 中（代行文化が未成熟） | **中** |
| CRM・リピーター | ★★★★☆ | ★★★★☆ | **大（未導入51.2%）** | **高** |
| データ分析 | ★★★☆☆ | ★★☆☆☆ | 大（必要性は感じるが行動せず） | **中** |
| SNS運用 | ★★★★☆ | ★★★☆☆ | 中 | **低**（代行は高額） |

**結論**: CRM/リピーター施策と商品コンテンツの2領域が、「課題の深刻度 × 支払意欲 × 競合の少なさ」の掛け算で最も参入機会が大きい。

## 2. ソリューション案：3段階のサービス設計

### 全体コンセプト

**「EC事業者が自分で判断できるようになる」ための AI活用支援パートナー**

単なるツール提供でも、丸投げ代行でもない。**「AIを使って自分のストアを理解し、次のアクションを自分で決められる状態」**を作ることが価値。

コンサル不信（シミュレーションの20%しか成果出ず、マージン50%中抜き等）が構造的に存在するため、**「結果が見える前にお金を取らない」**設計が信頼構築の鍵。

### Step 1: 無料AI診断（リードジェネレーション）

**目的**: 信頼構築 + 課題の可視化 + 有料サービスへの導線

**提供内容**:
- Shopifyストアに接続し、AIが自動で「改善ポイントレポート」を生成
- 所要時間: 事業者側10分（接続許可のみ）、自動分析30分

**レポート内容**:
1. **商品ページスコア**: 説明文の充実度、SEO最適化度、画像品質を10段階で評価
2. **売上パターン分析**: 直近30日の売上推移、AOV、商品別パフォーマンス
3. **リピート率診断**: 新規 vs リピーターの比率、LTV概算
4. **改善提案（上位3つ）**: AIが特定した最もインパクトの大きい改善ポイント

**技術的実現性**: Phase 3の調査から、すべてShopify Admin API + Claude APIで実現可能（Products, Orders, Customers APIの組み合わせ）

**この段階で収集できるデータ**:
- 事業者の実際の商品数、売上規模、顧客構成
- 「どの改善提案に興味を示したか」= 真の課題の特定

- **開発内容**: AI商品説明文一括生成機能 + Claude Visionパイプライン
- **工数**: 約15時間
- **成果物**: クライアント向けの実行ツール

### Month 2: CRM連携 + ダッシュボード
- **開発内容**: Klaviyo API連携 + Looker Studioテンプレート + RFM分析
- **工数**: 約20時間
- **成果物**: 月額サービスのコアインフラ

### Month 3-4: 自動化の深化
- **開発内容**: 週次AIレポート自動生成 + Webhook連携 + 在庫アラート
- **工数**: 約20時間
- **成果物**: 月額サービスの運用コストを最小化する自動化基盤

## 5. リスクと対策

### リスク1: Shopify Magicの進化により差別化が消失
- **対策**: ツール提供ではなく「活用コンサル」に軸足を置く。Shopify Magicが進化すればそれを使いこなす支援に転換
- **モニタリング**: Shopify EditionsやChangelog を月次チェック

### リスク2: 最初の有料クライアントが獲れない
- **対策**: 初期2〜3社は成果報酬型（成果が出なければ無料）で実績を作る
- **Phase 4のチャネル設計**に従い、7日間で3〜5名のインタビュー候補を確保

### リスク3: 工数が見積もりを超過（Day 3検証で1.7〜2.7倍の超過が判明済み）
- **対策**:
  - MVPは2機能に絞り込み済み（商品ページ + 売上分析）
  - Claude Code活用で開発工数を圧縮
  - 月額サービスのAI自動化率を最大化し、運用工数を月5〜8時間に抑える

### リスク4: コンサル不信による拒絶
- **対策**:
  - 「コンサルタント」を名乗らない。「AI活用パートナー」
  - 無料診断で先に価値を見せる
  - Phase 1で判明した失敗パターン（シミュレーション詐欺、担当者変更等）を**明示的に否定**する営業トーク
  - 「私1人でやっているので担当者は変わりません」が個人フリーランス最大の武器

## 6. 検証すべき仮説（インタビューで確認）

本ソリューション案の前提となる仮説のうち、**リアルな顧客との対話でしか検証できない**もの：

| # | 仮説 | 検証方法 | 判断基準 |
|---|------|---------|---------|
| H1 | 商品ページの説明文改善に¥30,000払う意欲がある | 価格感度質問 | 5名中2名以上が「払う」 |
| H2 | CRM未導入の理由は「難しそう」であり「不要」ではない | 直接質問 | 未導入者の過半数が「やりたいが...」 |
| H3 | 無料AI診断レポートを見れば有料サービスに興味を持つ | 診断デモ後の反応 | 5名中3名以上が「詳しく聞きたい」 |
| H4 | 月額¥80,000は月商500〜1,000万円の事業者にとって許容範囲 | 価格感度質問（Van Westendorp法） | 「高いが検討する」以上が過半数 |
| H5 | 「AI活用パートナー」という呼称はコンサル不信を回避できる | 反応観察 | 「コンサル」との比較で好意的反応 |
| H6 | 事業者が最も改善したい指標はCVR/リピート率/AOVのいずれか | 優先順位質問 | 一致度の高い指標を特定 |

## まとめ

**ソリューションの核心**: 「AIで時間を作る」のではなく、「AIで正しい判断ができるようになる」

- **無料診断** → 信頼構築 + 課題の可視化
- **スポット施策** → 小さな成功体験の提供
- **月額伴走** → 継続的な改善サイクルの構築

この3段階ファネルにより、「コンサルに騙された」トラウマを持つEC事業者にも、リスクなく始められる入口を用意する。

## インタビューの目的

## スクリプト本体

### 0. アイスブレイク（2分）

```
本日はお忙しい中お時間いただきありがとうございます。
[名前]と申します。EC事業者の方々が日常的にどんな課題を
感じていらっしゃるか、率直なお話を聞かせていただければと思っています。

売り込みは一切ありません。純粋にお話を聞かせていただく場です。
もし差し支えなければ、メモのために録音させていただいてもよろしいですか？

最初に、かんたんに自己紹介をお願いできますか？
どんな商品を扱っていらっしゃるか、ざっくりで大丈夫です。
```

**聞き取りポイント**: 商品カテゴリ、月商規模（直接聞かない、自然に出てくるのを待つ）、運営年数、スタッフ数（1人か複数か）

---

### 1. 日常業務の実態（8分）

**目的**: 時間の使い方のリアルな実態を把握する

#### Q1-1: 典型的な1日の流れ
```
ECストアの運営で、典型的な1日の流れを教えていただけますか？
朝起きてから、どんな順番でどんな作業をされていますか？
```

**深掘りポイント**:
- 「それにはだいたいどのくらい時間がかかりますか？」
- 「その作業は毎日やるものですか？週1くらいですか？」
- 「その中で一番"これやりたくないな"と思う作業はどれですか？」

#### Q1-2: 最も時間がかかっている作業
```
EC運営の中で、「もっと楽にならないかな」と一番感じる作業は何ですか？
```

**深掘りポイント**:
- 「具体的にどの部分が大変ですか？」（例: 商品登録なら撮影？テキスト？登録操作？）
- 「それを解決するために、これまで何か試したことはありますか？」
- 「それはうまくいきましたか？」

#### Q1-3: 想定外の発見を探る質問
```
EC運営を始める前と比べて、「思っていたより大変だった」ことはありますか？
```

---

### 2. お金の使い方（7分）

**目的**: 支払意欲と価格感度を探る（H1, H4の検証）

#### Q2-1: 現在の外部サービス利用状況
```
今、EC運営のために月々いくらくらいツールやサービスに払っていますか？
Shopifyの月額以外で、使っているツールやサービスがあれば教えてください。
```

**深掘りポイント**:
- 各ツール/サービスの名前、月額、満足度
- 「やめようと思ったツールはありますか？なぜですか？」
- 「逆に、"これには助けられている"というツールはありますか？」

#### Q2-2: 外注の経験
```
作業の一部を外注したことはありますか？
例えば、商品撮影、ページ制作、広告運用、コンサルなど。
```

**深掘りポイント（外注経験ありの場合）**:
- 「いくらくらいでしたか？」
- 「期待通りの結果でしたか？」
- 「次も同じサービスを使いますか？なぜ？」

**深掘りポイント（外注経験なしの場合）**:
- 「外注を検討したことはありますか？」
- 「外注しない理由は何ですか？」（コスト？信頼？自分でやりたい？）

#### Q2-3: 価格感度（H1, H4）
```
仮に、AIを使ってストアの商品ページを20〜30点まとめて改善してくれるサービスが
あったとして、いくらくらいなら試してみようと思いますか？

逆に、「この金額以上は絶対払わない」というラインはどのあたりですか？
```

**Van Westendorp法の4質問**（時間があれば）:
1. 「いくらから"安すぎて不安"だと感じますか？」
2. 「いくらから"安くてお得"だと感じますか？」
3. 「いくらから"高いけど検討する"と感じますか？」
4. 「いくらから"高すぎて無理"と感じますか？」

### 3. データ活用とCRM（5分）

**目的**: CRM/データ分析の実態と障壁を把握（H2, H6の検証）

#### Q3-1: データの見方
```
売上のデータは普段どうやって見ていますか？
Shopifyの管理画面？GA4？それとも特に見ていない？
```

**深掘りポイント**:
- 「そのデータを見て、何かアクションを変えたことはありますか？」
- 「"この数字が分かればいいのに"と思うことはありますか？」

#### Q3-2: リピーター施策（H2）
```
リピーターのお客様を増やすために、何か施策はされていますか？
例えば、メルマガ、LINE、ポイント制度など。
```

**深掘りポイント（やっている場合）**:
- 「どのくらいの頻度で配信していますか？」
- 「効果は感じていますか？」
- 「一番大変なのはどの部分ですか？」

**深掘りポイント（やっていない場合）**:
- 「やった方がいいとは思いますか？」
- 「やらない理由は何ですか？」→ **ここがH2の核心**
  - a) 時間がない
  - b) やり方が分からない
  - c) ツールの使い方が難しい
  - d) 効果が分からないから優先度が低い
  - e) そもそも必要性を感じない

#### Q3-3: 最も改善したい指標（H6）
```
もし1つだけ数字を改善できるとしたら、何を改善したいですか？

1. 新規のお客様の数
2. 1回あたりの注文金額
3. リピート率
4. 商品ページの購入率（CVR）
5. その他
```

### 4. AI活用と将来の展望（5分）

**目的**: AI活用の認知度と期待値を把握（H3, H5の検証）

#### Q4-1: AI利用状況
```
ChatGPTやAIツールをEC運営で使ったことはありますか？
```

**深掘りポイント（使っている場合）**:
- 「具体的にどんな場面で使っていますか？」

**深掘りポイント（使っていない場合）**:
- 「使ってみたいとは思いますか？」
- 「何に使えたら嬉しいですか？」

#### Q4-2: AI診断への反応（H3）

もし無料で、AIがあなたのShopifyストアを自動分析して、
「ここを改善すれば売上が上がる可能性が高い」というレポートを
出してくれるサービスがあったら、使ってみたいと思いますか？
```

**深掘りポイント**:
- 「どんな情報が含まれていたら"使ってみたい"と思いますか？」
- 「逆に、"怪しいな"と思うポイントはありますか？」
- 「ストアのデータを連携することに抵抗はありますか？」

#### Q4-3: サービス呼称への反応（H5）
```
こういったサービスを提供する人がいるとして、
「ECコンサルタント」と「AI活用パートナー」では、
どちらの方が相談しやすいと感じますか？
```
